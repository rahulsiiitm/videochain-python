\section{Introduction}

This chapter defines the formal requirements that guided the design and development of VidChain. The process began with identifying who would actually use the system, what problems they needed solved, and what constraints the hardware environment imposed. Requirements are classified using the MoSCoW method (Must Have, Should Have, Could Have, Won't Have) and are divided into functional and non-functional categories. Hardware constraints are derived directly from the project's technical specifications.

\section{User Personas}

\subsection{The Academic Researcher}

The primary persona for this system is an academic researcher working in fields such as computer vision, human-computer interaction, or behavioural analysis. This persona typically works with recorded video data, including lecture recordings, lab experiment footage, and field study captures, and needs to extract structured, queryable information from long sessions without manually reviewing every minute of footage.

\textbf{Goals:}
\begin{itemize}
    \item Extract timestamped events, spoken dialogue, and on-screen text from research recordings.
    \item Query video content using natural language rather than manual scrubbing.
    \item Generate structured summaries suitable for inclusion in research notes or reports.
    \item Run analysis locally without uploading sensitive research data to external cloud services.
\end{itemize}

\textbf{Pain Points:}
\begin{itemize}
    \item Existing tools require either cloud APIs (raising data privacy concerns) or server-grade hardware that is inaccessible to individual researchers.
    \item Standard video players offer no semantic search capability, so finding a specific moment requires manual effort.
    \item Most AI-powered video tools produce flat, unstructured outputs with no temporal continuity between observations.
\end{itemize}

\textbf{Technical Comfort Level:} Moderate to high. Comfortable running a command-line tool or a local web interface, but does not want to configure deep infrastructure.

\textbf{Hardware Profile:} A personal workstation or university-issued laptop, typically equipped with a consumer NVIDIA GPU in the 4--12\,GB VRAM range (e.g., RTX 3050, RTX 4060). No access to dedicated compute clusters for personal projects.

\section{Functional Requirements}

Functional requirements describe what the system must do. They are organised using the MoSCoW prioritisation framework.

\subsection{Must Have}

\begin{itemize}
    \item \textbf{Video Ingestion.} The system must accept a local video file path and process it through the full multimodal pipeline, producing a structured, searchable knowledge base.

    \item \textbf{Multimodal Extraction.} The system must extract at minimum three independent data streams per video: visual scene descriptions (via VLM or YOLO), speech transcription (via Whisper), and on-screen text (via OCR).

    \item \textbf{Natural Language Query.} The system must allow the user to query ingested video content using free-form natural language and return a coherent, timestamped response.

    \item \textbf{Temporal Summarisation.} The system must be capable of producing a narrative summary of an entire video using the Recursive Map-Reduce pipeline, without exceeding LLM context window limits.

    \item \textbf{Persistent Storage.} All ingested video knowledge bases, vector stores, and knowledge graphs must persist to disk and survive server restarts.

    \item \textbf{Session Management.} The system must support multiple independent sessions, each isolated to a specific video context, with create, rename, and delete operations.

    \item \textbf{Local-First Execution.} The entire pipeline, including LLM inference, embedding, and retrieval, must run locally without requiring any external API calls, unless the user explicitly configures a cloud LLM.
\end{itemize}

\subsection{Should Have}

\begin{itemize}
    \item \textbf{GraphRAG Integration.} The system should construct a Temporal Knowledge Graph per video session, enabling multi-hop entity queries that flat vector search cannot resolve.

    \item \textbf{Web Dashboard.} The system should provide a browser-based interface exposing the chat, video player, semantic heatmap, and live telemetry without requiring a separate installation.

    \item \textbf{Confidence Scoring.} IRIS should self-evaluate and return a confidence score alongside each response, giving the user a signal of how well the answer is grounded in evidence.

    \item \textbf{Adaptive Keyframe Filtering.} The pipeline should skip visually redundant frames before they reach heavy compute nodes, reducing GPU load without sacrificing coverage.

    \item \textbf{Hardware Telemetry.} The system should expose live CPU, GPU, and VRAM utilisation during ingestion, visible through the dashboard's Neural HUD.
\end{itemize}

\subsection{Could Have}

\begin{itemize}
    \item \textbf{Emotion Analysis.} The system could include an optional \texttt{EmotionNode} (DeepFace) that classifies the dominant emotion of detected persons on a per-frame basis.

    \item \textbf{Action Classification.} An optional \texttt{ActionNode} (MobileNetV3) could supplement VLM scene descriptions with high-level activity labels (e.g., \textit{suspicious}, \textit{emergency}).

    \item \textbf{Desktop GUI.} A native desktop application (\texttt{vidchain-studio}) could provide an alternative interface for users who prefer a standalone tool over a browser.

    \item \textbf{Report Export.} The system could allow users to export the full conversation history and IRIS responses to a structured Markdown report with a single click.
\end{itemize}

\subsection{Won't Have (in v1.0.0)}

\begin{itemize}
    \item \textbf{Real-time Stream Ingestion.} Live video stream processing (RTSP, webcam) is out of scope for the current release. The pipeline is designed for pre-recorded files only.

    \item \textbf{Multi-user Collaboration.} Shared sessions or concurrent multi-user access are not supported. VidChain is a single-user, local-first tool.

    \item \textbf{Cloud Deployment.} Managed cloud hosting, containerised deployment, or SaaS distribution are explicitly deferred to a future release.
\end{itemize}

\section{Non-Functional Requirements}

Non-functional requirements define the quality attributes and operational constraints of the system.

\subsection{Performance}

\begin{itemize}
    \item The \texttt{AdaptiveKeyframeNode} must achieve at least 60\% frame reduction on static or low-motion footage compared to uniform frame sampling, preserving GPU budget for meaningful frames. Benchmarks from v1.0.0 confirm 60--80\% savings on surveillance-style content.

    \item The Moondream VLM must process each accepted keyframe in under 5 seconds on the minimum supported hardware configuration (4\,GB VRAM). This was validated at approximately 2--5 seconds per frame during development.

    \item The FastAPI server must respond to \texttt{/api/query} requests within 30 seconds under normal operating conditions on the minimum hardware profile.
\end{itemize}

\subsection{Reliability}

\begin{itemize}
    \item Every inference stage (Whisper, VLM, OCR, DeepFace, YOLO) must fail independently. A crash in one node must not terminate the pipeline; partial results must always be returned.

    \item The system must resolve video playback correctly for absolute local paths on Windows and Unix-based systems, handled by the VidChain Media Gateway (\texttt{/api/media-stream}).
\end{itemize}

\subsection{Privacy and Security}

\begin{itemize}
    \item No video data, transcripts, or query results may be transmitted to external servers unless the user has explicitly configured a remote LLM provider via \texttt{litellm}.

    \item Session deletion must trigger a full memory purge, including ChromaDB vectors, the associated \texttt{.pkl} knowledge graph, and the \texttt{.json} knowledge base archive.
\end{itemize}

\subsection{Usability}

\begin{itemize}
    \item A user must be able to ingest a video and receive a natural language summary with no more than two commands (\texttt{vidchain-serve} followed by a browser interaction), without writing any code.

    \item The web dashboard must be responsive and functional on screens with a minimum width of 768\,px (tablet and above).
\end{itemize}

\subsection{Maintainability}

\begin{itemize}
    \item All pipeline components must conform to the \texttt{BaseNode} interface, allowing new modalities to be injected into any \texttt{VideoChain} without modifying existing nodes.

    \item The codebase must maintain zero redundant legacy files; deprecated modules (\texttt{processor.py}, \texttt{audio\_loader.py}) are retained only as stub references and carry explicit deprecation notices.
\end{itemize}

\begin{verbatim}

This decision was made to avoid shipping an incorrect CUDA build on machines with different driver versions, which is a common failure point in edge-deployed AI systems.

\subsection{Edge Optimisation Strategy}

The 4\,GB VRAM constraint is not a limitation imposed on the system, it is a design target. Several architectural decisions exist specifically to honour it:

\begin{itemize}
    \item The \texttt{AdaptiveKeyframeNode} eliminates redundant frames before they reach the GPU, minimising peak VRAM pressure.
    \item Moondream was chosen as the default VLM over LLaVA 7B because it delivers comparable scene understanding at approximately 186$\times$ lower latency on 4\,GB hardware.
    \item The \texttt{EmotionNode} (DeepFace) is threaded on CPU explicitly to avoid competing with VLM/YOLO for VRAM.
    \item YOLO is kept as a fast-mode fallback via the \texttt{--fast} CLI flag for long videos where VLM throughput would be prohibitive.
    \item The \texttt{VisionEngine} forces \texttt{FP32} precision (\texttt{half=False}) to prevent layer-fusion crashes on RTX 30-series laptops, which exhibit a known FP16 instability under certain Ultralytics builds.
\end{itemize}

\section{Summary}

This chapter established the requirements boundary for VidChain v1.0.0. The single primary persona, the academic researcher, shaped the emphasis on local-first privacy, consumer hardware compatibility, and natural language accessibility. The MoSCoW classification separates the core intelligence pipeline (Must Have) from optional enrichment nodes (Could Have), providing a clear scope boundary. Non-functional requirements formalise the performance targets and reliability guarantees that were validated iteratively during development, and hardware constraints reflect real architectural trade-offs made to keep the system accessible on edge hardware.
