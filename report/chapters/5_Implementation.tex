\section{Introduction}

This chapter provides a code-level breakdown of the six core algorithmic components of VidChain. Each section describes not just \textit{what} the code does but \textit{why} it was implemented that way, tracing design decisions back to the architectural constraints established in Chapter 4. All implementations are presented as structured pseudocode to communicate logic clearly without being tied to Python syntax.

\section{Pipeline Execution: \texttt{VideoChain.run()}}

\subsection{Overview}

The \texttt{VideoChain.run()} method is the central execution loop of the entire system. It is responsible for decoding the video frame by frame, constructing a shared context dictionary per frame, passing that context sequentially through every registered node, and accumulating the results into a unified temporal timeline.

\subsection{Algorithm}

\begin{algorithm}[H]
\caption{\texttt{VideoChain.run(video\_path)}}
\begin{algorithmic}[1]
\Require $video\_path$: path to a local video file
\Ensure $timeline$: list of fused event dictionaries

\State Extract audio track to temporary \texttt{.wav} file
\State Open video capture; read $fps$ and $total\_frames$
\State Initialise $timeline \gets [\,]$, $frame\_idx \gets 0$

\While{frames remain in video}
    \State Read next frame $F$
    \If{$frame\_idx \bmod frame\_skip \neq 0$}
        \State $frame\_idx \mathrel{+}= 1$; \textbf{continue}
    \EndIf

    \State $t \gets frame\_idx \;/\; fps$
    \State Initialise context $C \gets \{\texttt{video\_path},\ \texttt{audio\_path},\ \texttt{frame}: F,\ \texttt{time}: t\}$

    \State $skip \gets \texttt{False}$
    \For{each $node$ in $self.nodes$}
        \State $C \gets node.\texttt{process}(C)$
        \If{$C[\texttt{skip\_frame}] = \texttt{True}$}
            \State $skip \gets \texttt{True}$; \textbf{break}
        \EndIf
    \EndFor

    \If{$\lnot\, skip$}
        \State Remove $C[\texttt{current\_frame}]$ \Comment{Free memory before storing}
        \State Append $C$ to $timeline$
    \EndIf

    \State $frame\_idx \mathrel{+}= 1$
\EndWhile

\State Close video capture
\State \Return $(timeline,\ audio\_path)$
\end{algorithmic}
\end{algorithm}

\subsection{Implementation Notes}

\textbf{Frame index vs.\ frame read.} The \texttt{frame\_skip} check is applied to $frame\_idx$ rather than skipping reads entirely. Every frame is read from the decoder, but only every $n$-th frame is processed. This is necessary because OpenCV's \texttt{cap.set(CAP\_PROP\_POS\_FRAMES)} seek is unreliable on certain codecs.

\textbf{Early abort on \texttt{skip\_frame}.} When the \texttt{AdaptiveKeyframeNode} sets \texttt{skip\_frame: True}, the loop breaks immediately without invoking any subsequent node, achieving the 60--80\% compute saving reported in Chapter 3.

\section{Adaptive Keyframe Filtering}

\subsection{Overview}

The \texttt{AdaptiveKeyframeNode} acts as a computational firewall. Its purpose is to detect whether the current frame is visually distinct enough from the previous one to justify running the downstream heavy models.

\begin{algorithm}[H]
\caption{\texttt{AdaptiveKeyframeNode.process(context)}}
\begin{algorithmic}[1]
\Require Context $C$ containing \texttt{current\_frame} $F$
\Ensure $C[\texttt{skip\_frame}]$ set appropriately

\State $G \gets \texttt{GaussianBlur}(\texttt{Grayscale}(F),\ kernel=(21,21))$

\If{$prev\_gray = \texttt{None}$}
    \State $prev\_gray \gets G$
    \State $C[\texttt{skip\_frame}] \gets \texttt{False}$
    \State \Return $C$
\EndIf

\State $D \gets \texttt{AbsDiff}(prev\_gray,\ G)$
\State $nonzero \gets \texttt{CountNonZero}(D > diff\_cutoff)$
\State $change\_pct \gets (nonzero \;/\; |D|) \times 100$

\If{$change\_pct < change\_threshold$}
    \State $C[\texttt{skip\_frame}] \gets \texttt{True}$
\Else
    \State $prev\_gray \gets G$
    \State $C[\texttt{skip\_frame}] \gets \texttt{False}$
\EndIf

\State \Return $C$
\end{algorithmic}
\end{algorithm}

\subsection{Implementation Notes}

\textbf{Gaussian blur differencing.} A $21\times21$ Gaussian blur removes high-frequency noise before the difference is computed, ensuring that the change percentage reflects genuine scene motion rather than sensor jitter.

\textbf{Reference update policy.} The reference frame is only updated when a genuine keyframe is accepted. This means change is measured relative to the last \textit{meaningful} frame, preventing drift on gradually shifting scenes.

\section{GraphRAG: Build and Query}

\subsection{Build Algorithm}

\begin{algorithm}[H]
\caption{\texttt{TemporalKnowledgeGraph.build\_from\_timeline(timeline)}}
\begin{algorithmic}[1]
\Require $timeline$: list of fused event dictionaries
\Ensure Directed graph $G$ populated with temporal entity facts

\For{each event $e$ in $timeline$}
    \State $t \gets e[\texttt{time}]$
    \State Add timestamp node $t\_node \gets \texttt{"t:"}t$ to $G$

    \For{each tracking string $s$ in $e[\texttt{tracking}]$}
        \State $entity \gets \texttt{ExtractEntityID}(s)$
        \If{$entity \neq \texttt{None}$}
            \State \texttt{AddOrUpdateEntityNode}$(G,\ entity,\ t)$
            \State \texttt{AddEdge}$(G,\ t\_node \to entity,\ relation=\texttt{detects})$
        \EndIf
    \EndFor

    \If{$|e[\texttt{objects}]| < 120$}
        \For{each YOLO object $obj$ in \texttt{ParseYOLO}$(e[\texttt{objects}])$}
            \State \texttt{AddOrUpdateEntityNode}$(G,\ obj,\ t)$
        \EndFor
    \EndIf

    \If{$e[\texttt{ocr}] \neq \texttt{None}$}
        \State \texttt{AddOCRNode}$(G,\ e[\texttt{ocr}],\ t)$
        \State \texttt{AddEdge}$(G,\ t\_node \to ocr\_node,\ relation=\texttt{screen\_shows})$
    \EndIf
\EndFor
\end{algorithmic}
\end{algorithm}

\subsection{Implementation Notes}

\textbf{The 120-character guard.} This threshold distinguishes between short structured YOLO output and free-form VLM prose, ensuring only structured entities are indexed into the graph rather than unstructured scene descriptions.

\begin{algorithm}[H]
\caption{Recursive Narrative Map-Reduce}
\begin{algorithmic}[1]
\Procedure{GenerateSummary}{$\mathcal{T},\ mode$}
    \State $chunks \gets \texttt{ChunkByWordLimit}(\mathcal{T},\ 1500)$
    \State $chapter\_summaries \gets \texttt{MapPhase}(chunks)$
    \State \Return \Call{RecursiveReduce}{$chapter\_summaries,\ mode$}
\EndProcedure

\Statex

\Procedure{RecursiveReduce}{$\mathcal{S},\ mode$}
    \If{$|\mathcal{S}| = 1$}
        \State \Return $\texttt{FinalPolish}(\mathcal{S}[0],\ mode)$
    \EndIf
    \State $\mathcal{G} \gets \texttt{Group}(\mathcal{S},\ batch\_size=5)$
    \State $\mathcal{R} \gets [\,]$
    \For{each batch $g \in \mathcal{G}$}
        \State $\mathcal{R}.\texttt{append}\bigl(\texttt{LLM}(\textit{Prompt}_{reduce},\ g)\bigr)$
    \EndFor
    \State \Return \Call{RecursiveReduce}{$\mathcal{R},\ mode$}
\EndProcedure
\end{algorithmic}
\end{algorithm}

\section{RAG Engine and Session Management}

\subsection{Session Lifecycle Algorithm}

\begin{algorithm}[H]
\caption{Session Lifecycle: Ingest and Delete}
\begin{algorithmic}[1]

\Statex \textbf{Ingest}
\State Lock $session[\texttt{video\_id}] \gets video\_id$
\State Lock $session[\texttt{video\_path}] \gets video\_source$
\State Persist session record to disk
\State Dispatch \texttt{\_background\_ingest()} as async background task

\Statex

\Statex \textbf{Delete}
\State $video\_id \gets session[\texttt{video\_id}]$
\State $\texttt{ChromaDB.delete}(where=\{\texttt{video\_id}\})$
\State Delete \texttt{knowledge\_graphs/graph\_\{video\_id\}.pkl}
\State Delete \texttt{knowledge\_bases/\{video\_id\}.json}
\State Delete \texttt{sessions/\{session\_id\}.json}
\end{algorithmic}
\end{algorithm}

\subsection{Implementation Notes}

\textbf{Thermal mitigation.} A $10\,\text{ms}$ sleep interval in the processing loop gives the RTX 3050 breathing room between frames, preventing sustained thermal load from triggering clock-speed throttling.

\textbf{VRAM-aware lazy loading.} Node initialisation is staggered by 2 seconds to ensure the CUDA context is stable before the next model claims its VRAM allocation.

\textbf{Three-artefact purge.} Every session deletion removes the ChromaDB collection entry, the pickled knowledge graph, and the JSON knowledge base in a single atomic sequence, guaranteeing that no residual data from a deleted session can influence future queries.

\section{Summary}

This chapter has presented the implementation of all six core algorithmic components of VidChain. Each was examined at the level of its controlling algorithm and then analysed for the non-obvious implementation decisions that distinguish it from a naive approach. The \texttt{VideoChain.run()} loop's sequential read strategy, the keyframe node's reference-frame update policy, the graph builder's VLM-prose guard, the summariser's word-budget chunking, the retrieval pipeline's post-rerank chronological re-sort, and the session manager's three-artefact purge each represent a deliberate engineering choice made in response to a concrete failure mode encountered during development. Chapter 6 presents the empirical results that validate these decisions.
